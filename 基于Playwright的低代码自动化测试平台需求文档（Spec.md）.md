# 基于 Playwright 的低代码自动化测试平台需求文档（Spec.md）

# 1. 文档概述

## 1.1 文档目的

本文档定义一套基于 Playwright 的低代码 Web 自动化测试平台需求，覆盖元素采集、元素库、用例编排、执行调度、报告归档、权限安全、数据存储和运维部署等内容。本文档用于指导产品设计、架构设计、开发实现、测试验收和后续迭代。

本次修订采用如下核心方案：

- 客户端元素采集采用“本地轻量 Agent + Playwright 启动浏览器 + 页面注入采集脚本 + 平台端审核入库”。
- 平台后端采用 MySQL 存储结构化业务数据。
- Redis 用于任务队列、录制会话、执行状态、分布式锁、限流和短期缓存。
- Playwright 执行能力从业务 API 服务中拆分为独立 Worker，避免浏览器进程影响主服务稳定性。

## 1.2 文档范围

文档覆盖以下模块：

- 用户、角色和权限管理。
- 项目、被测系统和环境管理。
- 本地 Agent 管理与元素采集。
- 元素库、页面对象和组件对象管理。
- 可视化测试用例编排。
- Playwright 执行任务、队列、并发控制和失败重试。
- 测试报告、截图、视频、trace 和日志归档。
- MySQL 数据模型和 Redis keyspace 规划。
- 安全、性能、兼容性、部署和验收标准。

首版不强制实现复杂插件市场、JIRA/Jenkins 集成、报告公开分享、完整审核流和多执行节点调度；这些能力作为后续版本增强项。

## 1.3 预期读者

产品经理、前端工程师、后端工程师、测试工程师、测试开发工程师、运维工程师、项目负责人。

## 1.4 术语定义

- SUT：System Under Test，被测 Web 系统。
- 本地 Agent：安装在测试人员本机的轻量服务，用于启动 Playwright 浏览器、采集元素、校验 locator 和上传录制数据。
- Playwright Worker：独立执行自动化测试任务的服务进程，负责浏览器生命周期、用例回放和执行产物生成。
- 元素定位 locator：Playwright 识别页面元素的表达方式，例如 test id、role、label、placeholder、text、CSS、XPath。
- 候选 locator：系统为同一个元素生成的多个可选定位方式，按稳定性、唯一性和可操作性评分。
- 元素库：保存页面、组件和元素信息的结构化资产库。
- 步骤 DSL：平台内部描述测试步骤的 JSON 数据结构，用于低代码编排、执行和报告展示。
- 执行产物：执行过程中生成的日志、截图、视频、trace、HTML 报告、Excel 报告等文件。

# 2. 项目背景与目标

## 2.1 项目背景

当前自动化测试存在如下痛点：

- 测试人员直接编写 Playwright 脚本门槛较高。
- 页面元素定位分散在脚本中，维护成本高。
- 登录、搜索、下单等公共步骤重复编写，复用不足。
- 自动化执行结果、失败截图、视频和 trace 缺乏统一归档。
- 多人协作时缺少统一的元素资产、用例资产和执行任务管理。

因此需要建设一套低代码自动化测试平台，通过元素采集、元素库、用例编排、执行调度和报告归档形成闭环。

## 2.2 项目目标

- 降低门槛：测试人员可以通过录制、拾取、拖拽和配置完成常见 Web 自动化用例。
- 提升复用：将页面、组件、元素、步骤模板沉淀为可复用资产。
- 提升稳定性：优先使用 test id、role、label 等稳定 locator，保存候选 locator，并支持有效性校验。
- 提升排障效率：每次执行自动归档日志、截图、视频、trace 和报告。
- 提升可维护性：业务 API、执行 Worker、文件存储和队列解耦。
- 控制首版范围：优先交付核心闭环，后续再扩展审核、集成、插件和多节点调度。

# 3. 总体架构

## 3.1 架构原则

- API 服务不直接长时间运行浏览器，浏览器进程由本地 Agent 或 Playwright Worker 管理。
- 录制数据先进入临时录制会话，用户审核、命名、分组、去重后再正式入库。
- 用例步骤统一存储为 JSON DSL，避免后续扩展时频繁修改表结构。
- 执行任务异步化，所有执行请求先落库，再进入 Redis 队列，由 Worker 消费。
- 执行产物不直接塞进数据库，数据库只保存元数据和文件地址。
- 密码和敏感变量不明文存储，报告和日志默认脱敏。

## 3.2 逻辑架构

```text
平台 Web 前端
  |-- 用户/项目/元素库/用例编排/报告查看
  |
后端 API 服务
  |-- 认证授权、业务数据、任务创建、报告索引
  |-- MySQL
  |-- Redis
  |
本地 Agent
  |-- Playwright 浏览器录制
  |-- 页面注入采集脚本
  |-- locator 生成与即时校验
  |
Playwright Worker
  |-- 消费 Redis 执行队列
  |-- 独立 browser context 执行用例
  |-- 生成截图、视频、trace、日志和报告
  |
对象存储/文件存储
  |-- 截图、视频、trace、HTML 报告、Excel 报告
```

## 3.3 技术栈要求

- 前端：React 或 Vue，支持可视化拖拽编排和实时执行状态展示。
- 后端 API：Node.js、Java 或 Python 均可，需提供 REST API 和 WebSocket/SSE。
- 本地 Agent：推荐 Node.js + Playwright，可封装为 CLI、Electron 或 Tauri 桌面程序。
- 执行 Worker：推荐 Node.js + Playwright Test Runner。
- 数据库：MySQL 8.0 及以上。
- 缓存和队列：Redis 6.2 及以上。
- 文件存储：本地文件系统、MinIO、阿里云 OSS 或 S3 兼容存储。

# 4. 核心业务流程

## 4.1 元素采集流程

```text
用户登录平台
  -> 创建录制会话
  -> 平台唤起本地 Agent
  -> Agent 使用 Playwright 启动 Chrome/Edge/Chromium
  -> Agent 访问被测系统 URL
  -> 注入采集脚本并监听用户操作
  -> 生成操作事件和候选 locator
  -> 使用 Playwright 即时校验唯一性、可见性、可操作性
  -> 上传到平台临时录制会话
  -> 用户审核、命名、分组、去重
  -> 保存到元素库或生成用例步骤
```

## 4.2 用例编排流程

```text
选择项目和环境
  -> 选择页面/组件/元素
  -> 拖拽生成步骤
  -> 配置动作、输入参数、变量、断言、等待策略
  -> 保存为测试用例
  -> 可选保存为步骤模板
```

## 4.3 执行流程

```text
用户发起执行
  -> API 创建 execution_job
  -> 任务进入 Redis 队列
  -> Worker 消费任务
  -> 创建独立 browser context
  -> 按步骤 DSL 执行
  -> 逐步写入执行状态、日志和产物
  -> 生成报告
  -> 更新任务完成状态
```

# 5. 功能需求

## 5.1 用户、角色和权限

平台内置三类角色：

- 管理员：拥有系统配置、用户管理、项目管理、执行配置和全部数据权限。
- 测试负责人：拥有项目内元素库、用例、执行、报告和成员查看权限。
- 测试人员：拥有元素采集、用例编排、用例执行和报告查看权限。

安全要求：

- 密码必须使用 bcrypt、argon2id 或 PBKDF2 等密码哈希算法，不允许使用 MD5。
- “记住密码”改为“记住登录状态”，客户端不得保存明文密码。
- 敏感变量、被测系统密码和 token 必须加密存储或接入密钥管理服务。
- 所有数据访问必须校验项目权限。

## 5.2 项目和环境管理

- 支持创建项目，项目对应一个或多个被测系统。
- 支持配置多个环境，例如本地、测试、预发、生产。
- 环境包含 base URL、环境类型、描述、是否允许执行、是否需要二次确认。
- 生产环境默认不允许执行破坏性用例，执行前必须有权限校验和二次确认。

## 5.3 管理台与项目协作需求

管理台采用“项目工作台”模式。用户登录后进入管理台首页，左侧固定展示菜单导航栏，右侧为当前功能工作区。所有录制、元素、用例、任务、报告等数据均以项目为边界进行管理，用户必须先选择或进入某个项目后再开展测试工作。

### 5.3.1 管理台整体布局

- 登录成功后进入管理台，不再以单页堆叠方式展示所有功能。
- 左侧导航栏固定展示一级菜单，建议包含：项目总览、录制管理、元素库、用例管理、执行任务、测试报告、项目成员、项目设置。
- 顶部区域展示当前项目、当前环境、当前登录用户、Agent 连接状态和常用操作入口。
- 主内容区域根据左侧菜单切换，保持项目上下文不丢失。
- 移动端或窄屏场景下，左侧导航栏可折叠为抽屉菜单。

### 5.3.2 项目维度管理

- 平台首页展示用户可访问的项目列表，包括项目名称、描述、负责人、环境数量、元素数量、用例数量、最近执行状态和更新时间。
- 用户进入项目后，所有业务菜单默认只展示当前项目的数据。
- 支持在管理台顶部快速切换项目，切换项目后录制会话、元素库、用例、执行任务和报告同步切换。
- 项目可配置默认环境、默认 base URL、默认浏览器、默认执行并发数、报告保留时间和录制策略。
- 项目删除默认采用软删除，存在成员、用例、报告或执行记录时不得直接物理删除。

### 5.3.3 项目成员协作

一个项目可以加入多个用户，用户在不同项目中可以拥有不同角色。系统级角色负责平台管理，项目级角色负责项目内协作。

项目内置角色建议如下：

- 项目管理员：管理项目设置、环境、成员、权限、元素、用例、任务和报告。
- 测试负责人：管理元素、用例、任务和报告，可查看项目成员。
- 测试人员：创建录制、维护元素、编排用例、发起执行和查看报告。
- 只读成员：仅可查看元素、用例、任务和报告，不可修改数据。

项目成员管理要求：

- 支持添加项目成员、移除项目成员、修改项目角色、禁用成员访问。
- 支持按用户名、昵称、邮箱搜索用户后加入项目。
- 同一用户在同一项目中只能存在一条有效成员关系。
- 成员被移除后，不删除其历史录制、用例、执行和报告记录，但后续不得继续访问该项目。
- 项目成员变更必须写入操作审计日志。

### 5.3.4 项目内模块导航

项目内菜单按测试资产生命周期组织：

- 项目总览：展示项目质量概况、最近录制、元素变化、用例通过率、执行趋势和失败 Top 列表。
- 录制管理：创建录制会话、查看录制事件、停止录制、回放录制、将录制内容转为元素和用例。
- 元素库：按页面、组件、元素层级展示采集结果，支持搜索、筛选、去重、审核、禁用和 locator 校验。
- 用例管理：按分组维护用例，支持录制转用例、拖拽编排、变量绑定、数据集和步骤模板。
- 执行任务：创建立即执行或批量执行任务，查看队列状态、运行状态、失败原因和重试记录。
- 测试报告：查看执行报告、失败证据、截图、视频、trace、趋势分析和导出结果。
- 项目成员：维护项目用户、项目角色、成员状态和最近活跃时间。
- 项目设置：维护项目信息、环境、执行策略、录制策略、报告保留策略和安全配置。

### 5.3.5 权限控制要求

- 所有项目数据接口必须校验当前用户是否属于该项目。
- 项目级权限应覆盖录制、元素、用例、任务、报告、成员和设置等模块。
- 只读成员不得触发录制、保存元素、修改用例、创建执行任务或变更项目配置。
- 测试人员不得管理项目成员和项目级安全配置。
- 测试负责人可管理测试资产，但默认不得删除项目、移除项目管理员或修改系统级配置。
- 管理员可进行系统级用户和角色管理，但访问项目数据仍应记录审计日志。

### 5.3.6 数据模型补充

项目成员关系建议新增 `tp_project_member` 表，用于维护用户和项目的多对多关系。

核心字段建议：

- `id`：主键。
- `project_id`：项目 ID。
- `user_id`：用户 ID。
- `project_role`：项目内角色，例如 `project_admin`、`test_lead`、`tester`、`viewer`。
- `status`：成员状态，例如 `active`、`disabled`、`removed`。
- `joined_at`：加入时间。
- `last_active_at`：项目内最近活跃时间。
- `created_by`：添加人。
- `created_at`、`updated_at`：创建和更新时间。

唯一性要求：`project_id` 和 `user_id` 的有效成员关系必须唯一。

### 5.3.7 验收标准

- 登录后可以看到左侧菜单导航栏，并能进入项目总览、录制、元素、用例、任务、报告、成员和设置模块。
- 用户只能看到自己有权限访问的项目。
- 进入任一项目后，各模块只展示当前项目的数据。
- 一个项目可以添加多个用户，同一用户可在不同项目中拥有不同项目角色。
- 项目成员权限变更后，用户刷新或重新登录即可按新权限访问。
- 元素库必须按页面和组件展示采集到的元素，能够区分元素所属项目、页面和组件。
- 所有项目成员新增、移除、角色变更和关键配置变更必须有审计日志。

## 5.4 本地 Agent 管理

- 平台提供 Agent 下载、安装说明和版本检测。
- Agent 启动后仅监听 `127.0.0.1`，不得暴露到公网或局域网。
- 平台通过一次性 token 与 Agent 建立录制会话。
- token 必须有过期时间，录制结束后立即失效。
- Agent 支持版本上报、心跳、能力上报和升级提示。

## 5.5 元素采集模块

采集方式分为两类：

- 操作录制模式：用户正常操作被测系统，系统自动采集操作步骤和对应元素。
- 元素拾取模式：用户开启拾取后，页面元素高亮，点击后只采集元素，不触发真实业务动作。

首版支持的动作：

- 页面跳转：goto。
- 点击：click、dblclick、right click。
- 输入：fill、clear、press。
- 选择：selectOption。
- 勾选：check、uncheck。
- 等待：等待元素可见、等待 URL、等待页面加载。
- 断言：元素存在、可见、文本匹配、URL 匹配、输入值匹配。

增强版再支持文件上传、下载、多标签页、iframe 深层操作、shadow DOM、拖拽、canvas 或复杂富文本、验证码和第三方登录处理。

## 5.6 locator 生成和校验

locator 优先级如下：

1. `data-testid`、`data-test`、`data-qa` 等测试属性。
2. role + accessible name。
3. label。
4. placeholder。
5. text。
6. 父子组合定位、filter、hasText。
7. 简短 CSS。
8. XPath 兜底，禁止默认生成纯层级 XPath。

采集时必须生成主 locator 和候选 locator 列表。每个候选 locator 记录 locator 类型、表达式、稳定性评分、是否唯一、是否可见、是否可操作、校验时间和失败原因。

## 5.7 元素库管理

元素库按以下层级组织：

```text
项目
  -> 页面
    -> 组件
      -> 元素
```

支持页面管理、组件管理、元素新增、编辑、禁用、删除、搜索、筛选、导入导出、有效性验证、元素去重和 locator 历史记录。元素删除默认采用软删除。若元素被用例引用，应提示影响范围，不允许直接物理删除。

## 5.8 用例编排模块

用例由步骤 DSL 组成，每个步骤包含：

- `action`：动作类型，例如 click、fill、select、assert、goto、wait。
- `target`：元素 ID 或临时 locator。
- `params`：动作参数。
- `timeout`：超时时间。
- `retry`：步骤级重试策略。
- `continueOnFailure`：失败后是否继续。
- `screenshotPolicy`：截图策略。
- `dataBinding`：变量绑定。

支持拖拽元素生成步骤、手动新增无元素步骤、步骤排序、复制、删除、禁用、变量引用、数据集执行、步骤模板和单步试运行。

## 5.9 执行模块

执行方式：

- 立即执行。
- 批量执行。
- 定时执行，作为 V1.1 能力。

执行参数包括项目、环境、base URL、浏览器、headless 模式、视口大小、并发数、失败重试次数、截图、视频和 trace 开关。

执行隔离要求：

- 每个执行任务使用独立 browser context。
- 任务结束必须回收浏览器、上下文和临时文件。
- 同一项目可配置最大并发数。
- Redis 分布式锁防止同一任务重复执行。
- Worker 异常退出时任务可标记为超时或重新入队。

浏览器支持策略：

- 录制首版支持 Chromium、Chrome、Edge。
- 执行首版支持 Chromium、Chrome、Edge。
- 增强版支持 Firefox 和 WebKit。
- Safari 不作为首版真实浏览器录制对象；Safari 兼容性通过 WebKit 近似验证。

## 5.10 报告模块

报告内容包括执行概况、环境信息、用例详情、失败证据、元素统计和日志片段。失败证据包括截图、视频、trace 中至少一种。

报告格式支持 HTML 在线查看、Excel 导出和 Playwright trace 链接。报告公开分享作为增强版能力。若启用，必须支持有效期、撤销、脱敏、访问审计和不可猜测 token。

## 5.11 系统配置

系统配置包含浏览器安装和版本检测、Playwright 版本管理、默认超时时间、重试次数、并发数、报告保存时间、日志保存时间、文件存储配置、Redis 队列配置、Worker 心跳和并发阈值。

# 7. 数据需求

## 7.1 MySQL 使用范围

MySQL 存储结构化、可审计、需长期保存的数据：

- 用户、角色、权限。
- 项目和环境。
- 页面、组件、元素和候选 locator。
- 录制会话和录制事件。
- 用例、步骤、版本、模板、变量和数据集。
- 执行任务、执行结果、步骤结果、执行产物和报告。
- 操作审计日志。

数据库名建议为：`test_platform`。

初始化脚本见：`database/init_mysql.sql`。

## 7.2 Redis 使用范围

Redis 不保存永久业务数据，用于短期状态和异步协调：

- 执行任务队列。
- 录制会话临时状态。
- Agent 心跳。
- Worker 心跳。
- 实时执行日志缓冲。
- 分布式锁。
- 接口限流。
- 登录会话或 token 黑名单。
- 元素校验结果短缓存。

Redis keyspace 规划见：`database/redis_keyspace.md`。

## 7.3 核心数据模型

核心表分组：

- 系统表：`sys_role`、`sys_user`、`sys_operation_log`。
- 项目表：`tp_project`、`tp_environment`。
- 元素资产表：`tp_page`、`tp_component`、`tp_element`、`tp_element_locator`、`tp_element_validation`。
- 录制表：`tp_recording_session`、`tp_recording_event`。
- 用例表：`tp_case_group`、`tp_test_case`、`tp_case_step`、`tp_case_version`、`tp_step_template`、`tp_variable`、`tp_dataset`。
- 执行表：`tp_execution_job`、`tp_execution_case_result`、`tp_execution_step_result`、`tp_artifact`、`tp_report`。

详细字段以 `database/init_mysql.sql` 为准。

# 8. 非功能需求

## 8.1 性能需求

- 页面普通查询响应时间 P95 ≤ 1 秒。
- 用例编排拖拽响应 P95 ≤ 300 毫秒。
- 录制事件上传延迟 P95 ≤ 1 秒。
- 执行状态推送延迟 P95 ≤ 1 秒。
- 首版支持 10 个用户同时在线、5 个执行任务并发。
- 单项目元素数量支持 5000 个，用例数量支持 2000 条，报告数量支持 10000 份。

## 8.2 稳定性需求

- API 服务连续运行 72 小时无崩溃。
- Worker 异常退出不影响 API 服务。
- 浏览器进程异常必须被回收。
- 执行任务状态必须可追踪，不能长时间停留在运行中。
- 执行失败必须保留错误原因和排障产物。

## 8.3 兼容性需求

- 平台 Web 前端支持 Chrome、Edge、Firefox 最新稳定版。
- 录制首版支持 Chromium、Chrome、Edge。
- 执行首版支持 Chromium、Chrome、Edge；增强版支持 Firefox 和 WebKit。
- Safari 不作为首版真实浏览器录制对象；Safari 兼容性通过 WebKit 近似验证。
- Agent 支持 Windows 10+、macOS 12+、Ubuntu 20.04+。

## 8.4 安全需求

- 密码不得使用 MD5。
- Agent 只监听 localhost，并使用一次性 token。
- 敏感变量加密存储，执行日志和报告默认脱敏。
- 报告、截图、视频和 trace 需要权限校验。
- 所有 URL 输入需要 SSRF 防护策略。
- 元素 locator、步骤参数和报告内容展示时需要 XSS 防护。
- 所有关键操作记录审计日志。

# 9. 部署与运维需求

## 9.1 部署组件

- Web 前端。
- API 服务。
- MySQL。
- Redis。
- Playwright Worker。
- 文件存储服务。
- 本地 Agent 安装包。

## 9.2 推荐部署方式

- 开发和测试环境使用 Docker Compose。
- 生产环境 API 和 Worker 分开部署。
- MySQL 和 Redis 使用独立实例。
- Worker 可横向扩展，但必须通过 Redis 队列和分布式锁控制任务。

## 9.3 监控要求

- API 响应时间、错误率、慢查询。
- Redis 队列长度、失败队列、延迟任务数量。
- Worker 心跳、并发数、任务失败率。
- 浏览器进程数量和资源使用率。
- 文件存储容量。

# 10. 验收标准

## 10.1 MVP 功能验收

MVP 必须完成：

- 用户登录和基础角色权限。
- 项目和环境管理。
- 本地 Agent 创建录制会话。
- 操作录制和元素拾取。
- 主 locator 和候选 locator 生成。
- 元素审核入库、分组、搜索、有效性验证。
- 可视化用例编排和步骤 DSL 保存。
- 执行任务入队、Worker 执行、并发限制和失败重试。
- 截图、视频、trace、日志和 HTML 报告归档。
- MySQL 表结构和 Redis keyspace 初始化。

## 10.2 指标验收

指标必须绑定指定样例系统、浏览器、测试数据和用例集合：

- 在指定样例系统中，录制生成唯一 locator 的元素比例 ≥ 90%。
- 在指定样例系统中，同一用例连续执行 10 次，非环境原因成功率 ≥ 95%。
- 单个执行任务失败时不影响其他任务继续执行。
- Worker 异常退出后任务状态可被恢复或标记为失败。
- 报告包含失败截图、视频或 trace 中至少一种排障产物。

## 10.3 非 MVP 延后项

以下功能不纳入 MVP 验收：

- 用例审核流。
- 用例版本回滚 UI。
- 报告公开分享。
- 报告对比。
- JIRA/Jenkins 集成。
- 自定义步骤插件。
- 多执行节点智能调度。
- 元素定期巡检任务。

# 12. 附录

## 12.1 元素采集推荐策略

平台首选“本地 Agent + Playwright + 页面注入采集脚本”的方式。该方案能访问企业内网系统，能复用 Playwright 的真实执行环境，也能即时做 locator 唯一性、可见性和可操作性校验。浏览器插件可作为增强体验，但不作为核心采集链路。

## 12.2 数据库与缓存维护

- MySQL 初始化脚本：`database/init_mysql.sql`。
- Redis keyspace 规划：`database/redis_keyspace.md`。
- 初始化执行脚本：`scripts/init_infra.py`。

初始化脚本必须幂等，不得删除现有业务数据。
